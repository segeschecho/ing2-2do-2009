  \end{itemize}
}
\frame
{
  \frametitle{La metáfora}
  \begin{itemize}
  \item Brooks compara un proyecto de software con un hombre lobo: algo inocente se transforma en un monstruo.
  \item Necesidad de una "bala de plata", algo que haga bajar costos y aumente productividad, confiabilidad y simplicidad.
  \item Mayor dificultad del sofware: la especificación, diseño y testing de la estructura conceptual.
  \end{itemize}
}
\frame
{
  \frametitle{Las dificultades}
  \begin{itemize}
  \item \emph{Dificultades Esenciales}: Inherentes a la naturaleza misma del software.
  	\begin{itemize}
  \item \textbf{Complejidad}: Intrínseca del software(escalabilidad, numeración de estados, comunicación).
  \item \textbf{Conformidad}: El software debe cumplir con limitaciones arbitrarias impuestas por personas y reglas de negocio.
  \item \textbf{Modificabilidad}: El software siempre va a estar sujeto a cambios.    
  \item \textbf{Invisibilidad}:El software es invisible e individualizable en el espacio. El software se intuye, pero no se ve.
  \end{itemize}
  \item \emph{Dificultades accidentales}: Dificultades no inherentes al software sino a su producción(Ej.Tipo de lenguaje de programación).
  \end{itemize}
}

\frame
{
  \frametitle{Avances que resolvieron dificultades accidentales}
  \begin{itemize}
  \item \textcolor{red}{Lenguajes de Alto Nivel}: Abstracciones conceptuales. Esconden complejidad accidental del programa compilado.
  \item \textcolor{red}{Time-Sharing}: La posibilidad de compartir el tiempo de ejecución entre procesos combate el accidente de
  los programas batch.
  \item \textcolor{red}{Ambientes de desarrollo unificado}: Combaten el accidente de tener aplicaciones que resuelven en forma
  individual las problemáticas comunes (bibliotecas integradas, formatos de archivos unificados, tuberías y filtros).
  \end{itemize}
}

\frame
{
  \frametitle{Esperanzas y potenciales balas de plata}
  \begin{itemize}
  \item \textbf{Lenguajes de alto nivel y POO}: Ventajas de subclasificación e information hiding.
  \item \textbf{Inteligencia Artificial y Sistemas Expertos}: Conjunto de reglas de base y motor de inferencia para facilitar el desarrollo a principiantes.
  \item \textbf{Programación Automática}: A partir de especificaciones generar código. Inviable y poco generalizable.
  \item \textbf{Programación Visual}: Inviable por la invisibilidad del software.
  \item \textbf{Verificación de Programas}: La verificación es costosa, no siempre aplica. Es suceptible a errores. Importancia de la validación.
  \item \textbf{Entornos y  herramientas de desarollo}: Facilitan el trabajo de los desarrolladores(Ej. reducen errores sintácticos). Postivo, pero contribución marginal.
  \end{itemize}
}

\frame
{
  \frametitle{Ataques a dificultades escenciales}
  \begin{itemize}
  \item\textbf{Comprar vs Contruir}: Si la construcción de software es tan difícil, entonces compremoslo!!.
  \item \textbf{Prototipos y refinación de requerimientos}: El cliente no sabe lo que quiere. Es muy importante el feedback.
  \item \textbf{Desarrollo iterativo incremental}: Es imposible contruir el producto en su totalidad de manera inmediata.
  \item\textbf{Buenos diseñadores}: Buen diseñador = buen diseño. Fomentar el crecimiento de buenos diseñadores.
  \end{itemize}
}

  \item \emph{Procesos iterativos incrementales(PU, Scrum).}
  \item \emph{Importancia de los buenos diseños.}
  \end{itemize}

  \end{itemize}
